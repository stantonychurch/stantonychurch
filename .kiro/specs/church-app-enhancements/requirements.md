# Requirements Document

## Introduction

This document specifies seven enhancement features for the St Antony Church devotional app — a Node.js/Express backend with a SQLite (better-sqlite3) database and a vanilla-JS single-page frontend. The enhancements cover full Tamil localisation, a media-rich Events Gallery, QR-code-based connection for Family Prayer Groups and Youth Corner, youth-targeted announcements, an admin database reset tool, and phone-number privacy for church members.

---

## Glossary

- **App**: The St Antony Church devotional web application.
- **Member**: A registered church member who authenticates through the Church Member portal.
- **Admin**: A privileged user who authenticates through the Church Admin portal (role = `admin`).
- **Youth Member**: A Member who has an `is_youth` flag set to `1` on their member record, assigned by an Admin.
- **Family Prayer Group**: A private group of Members connected via QR Code for shared prayer, songs, videos, and prayer reminders.
- **Youth Connect Group**: A private group of Members connected via QR Code for shared content and group messaging within the Youth Corner.
- **QR Code**: A machine-readable image generated by the App that encodes a unique group-join URL.
- **Join Token**: A cryptographically random, URL-safe string of at least 16 characters generated by the App and embedded in the group-join URL that uniquely identifies a specific group.
- **Event Folder**: A named album within the Events Gallery section, created by an Admin, that holds related event images and videos.
- **Media Item**: A single image or video file uploaded by an Admin and associated with an Event Folder.
- **Locale**: The active language setting of the App interface — `en` for English, `ta` for Tamil.
- **Translation Key**: A named string in the App's translation dictionary (`T`) used to render UI text.
- **Language_Switcher**: The UI control (the `lang-toggle` button) that changes the active Locale.
- **Database_Reset**: An Admin-initiated action that permanently deletes all user-generated data from the App database while preserving the default admin account and seeded content.
- **Member_Directory**: The list of registered Members visible to other Members within the App.
- **JWT**: JSON Web Token used to authorise authenticated API requests.
- **Target_Audience**: A field on an announcement record that restricts which Members can view the announcement (`all` or `youth`).

### Requirement 1: Tamil Language Full Localisation

**User Story:** As a Member, I want to switch the App to Tamil so that I can use the entire interface in Tamil without any English text remaining.

#### Acceptance Criteria

1. WHEN a Member activates the Tamil Locale via the Language_Switcher, THE App SHALL render all navigation labels, section headings, button text, form labels, placeholder text, and error messages using the Tamil Translation Keys defined in the `T.ta` dictionary, replacing every English value with its corresponding Tamil value in the UI.
2. WHEN a Member activates the Tamil Locale, THE App SHALL persist the selected Locale to `localStorage` under the key `lang` with the value `ta` so that the Tamil Locale remains active across page reloads without requiring the Member to re-select it.
3. WHEN a Member activates the Tamil Locale and a content item has a non-empty `_tamil` column value (e.g., `verse_tamil`, `title_tamil`, `content_tamil`), THE App SHALL display that Tamil column value instead of the English column value.
4. WHILE the Tamil Locale is active and a content item has no Tamil translation stored, THE App SHALL display the corresponding English column value as a fallback so that no field appears empty to the Member.
5. WHILE the Tamil Locale is active, THE Language_Switcher SHALL display the label "English" as the toggle label (indicating the user can switch back to English); WHILE the English Locale is active, THE Language_Switcher SHALL display the label "தமிழ்" as the toggle label.
6. WHEN a Member registers or logs in while the Tamil Locale is active, THE App SHALL display all authentication form labels, placeholder text, and error messages using the Tamil Translation Keys.
7. WHEN a Member activates the Tamil Locale, THE App SHALL update the `lang` attribute of the HTML root element to `ta`; WHEN a Member activates the English Locale, THE App SHALL update the `lang` attribute to `en`, so that the browser applies correct font rendering for each language.
8. WHEN a Member switches the Locale while a portal section is already rendered, THE App SHALL immediately re-render the active section's content and labels in the newly selected Locale without requiring the Member to navigate away and back.

### Requirement 2: Events Gallery with Folders and Media Uploads

**User Story:** As a Member, I want to browse past church event images and videos organised into named folders so that I can relive church memories easily.

**User Story:** As an Admin, I want to create event folders and upload images and videos into them so that I can organise and share church event media with all members.

#### Acceptance Criteria

1. THE App SHALL display an "Events Gallery" navigation entry on the Member home page that opens the Events Gallery section.
2. WHEN an Admin creates an Event Folder by providing a non-empty folder name of between 1 and 100 characters, THE App SHALL store the Event Folder record and display it as a named tile in the Events Gallery.
3. WHEN an Admin uploads an image file (JPEG, PNG, GIF, or WebP, verified by both MIME type and file extension, and no larger than 2 GB) to an Event Folder, THE App SHALL store the file in the server upload directory and associate the Media Item with that Event Folder.
4. WHEN an Admin uploads a video file (MP4, MOV, or WebM, verified by both MIME type and file extension, and no larger than 10 GB) to an Event Folder, THE App SHALL store the file in the server upload directory and associate the Media Item with that Event Folder.
5. IF an Admin attempts to upload a file whose MIME type or file extension does not match an accepted image or video type, or whose individual file size exceeds the applicable per-file limit, THEN THE App SHALL reject the upload with a descriptive error message and SHALL NOT store the file.
6. WHEN a Member opens the Events Gallery, THE App SHALL display all Event Folders as named tiles sorted by creation date with the most recent folder first.
7. WHEN a Member opens an Event Folder, THE App SHALL display all Media Items belonging to that folder; images SHALL render as thumbnails and videos SHALL render with a native playback control.
8. WHEN an Admin deletes an Event Folder that exists, THE App SHALL delete the Event Folder record, all associated Media Item records, and the associated physical files from the upload directory; IF the folder does not exist, THE App SHALL return a descriptive not-found error.
9. WHEN an Admin deletes a single Media Item that exists, THE App SHALL delete that Media Item record and its associated physical file from the upload directory; IF the Media Item does not exist, THE App SHALL return a descriptive not-found error.
10. IF an Admin submits a folder creation request without a folder name, or with a folder name exceeding 100 characters, THEN THE App SHALL return a descriptive error message and SHALL NOT create the folder.
11. THE App SHALL display the Media Item count inside each Event Folder tile so that Members know how many items to expect before opening the folder.

### Requirement 3: Family Prayer Group — QR Code Connect

**User Story:** As a Member, I want to create or join a Family Prayer Group by scanning a QR code so that my family members and I can share prayers, songs, and videos privately.

#### Acceptance Criteria

1. WHEN a Member navigates to the Family Prayer section, THE App SHALL display options to create a new Family Prayer Group and to join an existing group using a QR Code or join URL.
2. WHEN a Member creates a Family Prayer Group by providing a group name of between 1 and 50 characters, THE App SHALL generate a unique Join Token, store the group record with the creator as the first member, and display a QR Code that encodes the join URL for that group.
3. WHEN a Member scans the QR Code or visits the join URL containing a valid Join Token, THE App SHALL add the Member to the corresponding Family Prayer Group and display a confirmation message stating "Joined [group name] successfully."
4. IF a Member attempts to join a Family Prayer Group using an invalid or non-existent Join Token, THEN THE App SHALL display a descriptive error message and SHALL NOT add the Member to any group.
5. WHILE a Member is part of a Family Prayer Group, THE App SHALL allow the Member to post a prayer request of between 1 and 1000 characters that is visible only to members of that specific group.
6. WHILE a Member is part of a Family Prayer Group, THE App SHALL allow the Member to share a worship song with a title of up to 100 characters and lyrics of up to 5000 characters that is visible only to members of that specific group.
7. WHILE a Member is part of a Family Prayer Group, THE App SHALL allow the Member to share a video URL that starts with `http://` or `https://` and is visible only to members of that specific group.
8. WHILE a Member is part of a Family Prayer Group, THE App SHALL allow the Member to set a prayer reminder specifying a time in HH:MM (24-hour) format and a message of up to 280 characters that is stored against all members of that specific group.
9. WHEN a Member creates a prayer reminder within a Family Prayer Group, THE App SHALL store the reminder and deliver an in-app notification to each member of the group within 60 seconds of the specified reminder time.
10. IF a Member who is not part of a Family Prayer Group attempts to read or post group content, THEN THE App SHALL return an access-denied error message and SHALL NOT return any group data.
11. WHEN a Member views the Family Prayer section, THE App SHALL display a QR Code for each group the Member belongs to so that the Member can share the code with additional family members.
12. IF a Member who is already a member of a Family Prayer Group attempts to join the same group again using the Join Token, THEN THE App SHALL return a message indicating the Member is already in the group and SHALL NOT create a duplicate membership record.

### Requirement 4: Youth Corner — QR Code Connect

**User Story:** As a Youth Member, I want to connect with other youth members via QR code so that we can share content and communicate within the Youth Corner.

#### Acceptance Criteria

1. WHEN a Member navigates to the Youth Corner section, THE App SHALL display an option to create a new Youth Connect Group and an option to join an existing group via QR Code or join URL.
2. WHEN a Member creates a Youth Connect Group by providing a group name of between 1 and 60 characters, THE App SHALL generate a unique Join Token (cryptographically random, URL-safe, at least 16 characters), store the group record with the creator as the first member, and display a QR Code encoding the join URL for that group.
3. WHEN a Member scans the QR Code or visits the Youth Connect join URL containing a valid Join Token, THE App SHALL add the Member to the corresponding Youth Connect Group and display a confirmation message.
4. IF a Member attempts to join a Youth Connect Group using an invalid or non-existent Join Token, THEN THE App SHALL display a descriptive error message and SHALL NOT add the Member to any group.
5. IF a Member who is already a member of a Youth Connect Group attempts to join the same group again, THEN THE App SHALL return a message indicating the Member is already in the group and SHALL NOT create a duplicate membership record.
6. WHILE a Member is part of a Youth Connect Group, THE App SHALL allow the Member to post text messages of between 1 and 1000 characters visible only to members of that specific group.
7. WHILE a Member is part of a Youth Connect Group, THE App SHALL allow the Member to share a content item (image URL, video URL, or hyperlink) of up to 2048 characters in length visible only to members of that specific group.
8. WHEN a Member leaves a Youth Connect Group, THE App SHALL remove the Member from the group and revoke that Member's access to the group's content feed.
9. WHEN a Member views the Youth Corner section, THE App SHALL display a QR Code for each Youth Connect Group the Member belongs to so that the Member can invite additional Members.

### Requirement 5: Youth-Specific Announcements

**User Story:** As an Admin, I want to send announcements specifically to youth members so that youth-relevant news is separated from general announcements.

**User Story:** As a Youth Member, I want to see announcements intended for youth in a dedicated feed so that I do not miss relevant news.

#### Acceptance Criteria

1. WHEN an Admin creates an announcement, THE App SHALL provide a Target_Audience selector with the options "All Members" and "Youth Only".
2. WHEN an Admin publishes a "Youth Only" announcement, THE App SHALL store the announcement record with a `target_audience` value of `youth`.
3. WHEN an Admin publishes an "All Members" announcement, THE App SHALL store the announcement record with a `target_audience` value of `all`.
4. WHEN a Member whose `is_youth` flag is `1` views the Announcements section, THE App SHALL display both announcements with `target_audience = all` and announcements with `target_audience = youth`.
5. WHEN a Member whose `is_youth` flag is `0` or unset views the Announcements section, THE App SHALL display only announcements with `target_audience = all`.
6. THE App SHALL render a visible "Youth" badge label in the announcement header for every announcement whose `target_audience` value is `youth` so that Youth Members can distinguish youth announcements from general ones.
7. IF an Admin submits a request to publish an announcement whose title is absent, empty, or whitespace-only, or whose content is absent, empty, or whitespace-only, THEN THE App SHALL return a descriptive error message and SHALL NOT save the announcement record.
8. WHEN an Admin sets a Member's `is_youth` flag to `1` or `0`, THE App SHALL update that Member's record immediately within the same authenticated admin session so that subsequent announcement queries reflect the change.

### Requirement 6: Admin Dashboard — Reset Database

**User Story:** As an Admin, I want an option in the Admin Dashboard to reset the entire database so that I can clear all app data and start fresh.

#### Acceptance Criteria

1. THE Admin Dashboard SHALL display a "Reset Database" button that is accessible only to an authenticated Admin.
2. WHEN an Admin clicks the "Reset Database" button, THE App SHALL present a confirmation dialog that explicitly states "All user-generated data will be permanently deleted. This action cannot be undone." and requires the Admin to click a confirm button before any deletion proceeds.
3. WHEN an Admin confirms the database reset, THE App SHALL delete all records from the following tables: `members`, `videos`, `audio`, `events`, `prayer_requests`, `announcements`, `faith_journals`, `worship_songs`, `quiz_questions`, `articles`, `member_streaks`, `member_history`, and all extended-schema tables created by `database-ext.js`.
4. WHEN an Admin confirms the database reset, THE App SHALL re-seed the default admin account (name = `admin`, password = `admin123`) so that Admin login remains functional immediately after the reset.
5. WHEN the database reset completes successfully, THE App SHALL display a non-destructive in-dashboard status message confirming the reset was successful, and SHALL write a server-side log entry containing the timestamp and the initiating admin's name.
6. IF the database reset API endpoint receives a request without a valid Admin JWT, THEN THE App SHALL return an HTTP 403 status and SHALL NOT delete any data.
7. WHEN an Admin dismisses or cancels the confirmation dialog, THE App SHALL take no action and no data SHALL be deleted.
8. WHEN an Admin confirms the database reset, THE App SHALL re-seed the default Bible verses and devotionals (as defined in the seed logic in `database.js`) so that the App has baseline starter content immediately after the reset.

### Requirement 7: Member Directory — Hide Phone Numbers

**User Story:** As a Member, I want my phone number to be hidden from other members so that my personal contact information remains private.

#### Acceptance Criteria

1. WHEN a Member requests the Member Directory, THE App SHALL return each member's name, profile bio, birthday, and profile visibility status and SHALL NOT include the `phone` key in any record of the response payload.
2. WHEN a Member requests any member listing accessible to non-Admin users — including group member lists (Family Prayer Group members, Youth Connect Group members), chat participant lists, and directory endpoints — THE App SHALL NOT include the `phone` key in any record of the response payload.
3. WHEN an Admin requests the member list from the Admin Dashboard, THE App SHALL include the `phone` field in the response so that the Admin retains full member records for administration purposes.
4. WHEN a Member views another Member's profile card in any section of the App, THE App SHALL render the member's name and optional bio and SHALL NOT render the phone number field or any placeholder for it.
5. WHEN a Member views the Member's own profile card, THE App SHALL render the Member's own phone number so that the Member can verify the registered number.
6. IF a non-Admin API request includes a query parameter or filter that references the `phone` field, THEN THE App SHALL return an HTTP 400 status and SHALL NOT return any phone-based query results.
